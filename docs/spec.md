# Saleslog 仕様（公開版）

外回り営業の顧客管理と活動記録を1か所にまとめる、小型の業務CRMアプリケーション。

## 1. 解決する課題

外回り営業の現場では、顧客情報・訪問予定・訪問結果がノートや表計算に分散しがちで、
「今日どこを回るか」「最近訪問できていない顧客はどこか」を把握するのに手間がかかる。
Saleslog は顧客マスタと活動記録を1か所で扱い、ダッシュボードでフォロー漏れを可視化する。

## 2. スコープ

- 顧客管理: 登録・編集・削除・検索・絞り込み・ページネーション
- 活動記録: 訪問の登録・編集・削除、顧客への紐付け、予定/完了ステータス
- ダッシュボード: 件数サマリ、訪問推移グラフ、未記録フォロー・今日の予定の導線
- エリア別ボード: エリアごとの顧客カード一覧
- 認証とロール別アクセス制御: Clerk によるサインイン。manager は全データ、sales は自分の担当顧客・自分の活動記録のみ（範囲はサーバ側で強制）
- ユーザー管理（manager 限定）: メンバーの追加・役割変更・サインインアカウントの紐付け
- 認証モード切替: `AUTH_MODE=fixed` で認証なしのローカル実行を明示的に有効化。公開環境や reverse proxy 配下では使わない。未設定時と `clerk` 指定時は JWT 検証を要求
- Agent: 顧客詳細から商談準備・フォローアップ・失注リスク確認を実行し、根拠付き成果物と承認後の業務レコード保存を扱う
- 読み取り専用デモモード（書き込み操作と閲覧時の期限切れ根拠の自動消去を止める閲覧用設定）

## 3. 技術構成

- フロントエンド: React + TypeScript + Vite + React Router + Tailwind CSS + TanStack Query + React Hook Form + Zod + Storybook
- バックエンド: FastAPI + SQLAlchemy + Alembic + SQLite
- 認証: Clerk（フロント）+ PyJWT による JWKS 公開鍵検証（バックエンド）。認可ロールの正本は DB
- テスト: Vitest / pytest / Playwright
- グラフはライブラリを使わず SVG で自前描画

## 4. 品質基準（検査方法と記録方針）

- アクセシビリティ: axe による自動検査を全8画面で実施し、結果を docs/verification.md に記録する
- E2E: Playwright で主要導線のシナリオテストを実施する
- ユニット/API テスト: Vitest と pytest。カバレッジを計測し記録する
- 性能: Lighthouse を主要画面で計測し、計測日時点の値として記録する
- Agent: run作成、承認、冪等性、根拠スナップショット、SSE再接続、FTS5権限境界を自動テストで確認する

## 5. 画面構成

| 画面 | 概要 |
|---|---|
| ダッシュボード | 件数サマリ・訪問推移・フォロー導線 |
| 顧客一覧 | 検索・絞り込み・ページネーション（URL 同期） |
| 顧客詳細 | 顧客情報と訪問履歴・次回予定 |
| 顧客詳細 Agent | 商談準備・フォローアップ・失注リスク確認、根拠確認、承認操作 |
| 活動記録フォーム | 登録・編集（離脱防止・下書き保存） |
| 活動記録一覧 | 顧客横断の記録一覧・未記録フィルタ |
| エリア別ボード | エリアごとの顧客カード一覧 |
| ユーザー管理 | メンバーの追加・役割変更・サインインアカウント紐付け（manager 限定） |
| サインイン | 認証有効時の入口（Clerk） |

本書は実装の進行に合わせて更新する。
